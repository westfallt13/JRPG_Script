## 2026, June 19th ##
1. Started working in data/item_database/equipment/armor/armor_effects to figure out a solution for the overall data loading system for the game. This will involve parsing data from JSON files in each subfolder dynamically to allow for flexible and scalable memory management and efficient access to item data during gameplay. The goal is to create a system that can read the JSON files, store the data in a structured format (e.g., dictionaries or custom classes), and then allow for easy retrieval of this data when needed for UI display and combat calculations.
2. Built the central data-loading system in data/db_loader/. Three pieces: json_db.gd (class_name JsonDB) holds the reusable static JSON helpers (read_json, index_by, load_dir, load_collection) so the file-reading logic lives in exactly one place; db_loader.gd is the autoloaded DbLoader singleton that aggregates every category and loads each one lazily, caching it after first access to keep startup cheap; equipment_effects.gd (class_name EquipmentEffects) resolves an item's named effects into the actual effect data.
3. Converted every per-folder dict.*.gd / *_effects.gd from a hand-written placeholder dictionary into a small loader that reads its sibling JSON/ folder via JsonDB. The data now lives in JSON, so adding an item is just dropping a .json file in the right JSON/ folder — no code changes. Item folders use one file per item keyed by a name field; effect folders use a single collection file keyed by effect_name.
4. Fixed two things that would have broken at runtime: the DbLoader autoload in project.godot pointed at a dead uid (no file owned it) so I re-pointed it at the real db_loader.gd by path; and every res:// path was wrong because the project root is two levels up (repo root), so the real prefix is res://JRPG_Code/JRPG_Script/data/... — corrected everywhere. Also fixed the old `inv` instead of `var` syntax error in consumables.gd.
5. Implemented item -> effect linking. Each equipment item names its effect(s) in a per-type field that I kept as-is (weapons weapon_effect, armor armor_effects, accessories accessory_effects); the value is an effect name or list of names. EquipmentEffects has a LINKS table that routes each item category to the correct field AND the correct effects collection (each weapon type resolves against its own *_effects, while all armor shares armor_effects and all accessories share accessory_effects). One uniform set of functions regardless of equipment type. This is the foundation for applying effects to UI text, attribute math, damage calculations, and enemy status conditions later, with conditions filtering the resolved list on top.

Created empty-array starter JSON files for all the effect collections so they load cleanly until filled.

### Phase 3 — effect application pipeline ###
15. Gave effects SEMANTICS. Until now an effect was `{ effect_name, description, value, condition? }` — `value` was a bare number nothing read. Extended the schema with `kind`/`target`/`op`: keen_edge is `{kind: stat_mod, target: crit_chance, op: add, value: 0.15}`; low_hp_rage is `{... target: damage, op: mult, value: 0.25, condition: wearer_hp_pct < 0.3}`; armor's defense_boost = +10% defense (mult), magic_resistance = +30% magic_defense (mult). That covers the three starter kinds the plan wanted: flat add, percent mult, conditional mult. (Side fix: magic_resistance's description said "10%" but its value was 0.3 — now that the value actually drives math, I made the text say 30% to match.)
16. Built `data/db_loader/effect_resolver.gd` — `class_name EffectResolver`, the aggregation step. `aggregate(effects)` folds a condition-filtered effect list into per-target modifiers `{stat: {add, mult}}`; `apply(base, mod)` = `(base + Σadd) * (1 + Σmult)`; `derived(base, stat, effects)` is the one-shot. Put it in db_loader/ with the rest of the effects engine and made it operate on plain arrays (no Combatant dependency) so enemies/items can reuse it. Chose ADDITIVE percent stacking (two +25% = +50%, not +56.25%) — simpler to balance; documented it as a deliberate choice. It does NOT clamp; whoever consumes the number clamps if needed.
17. Wired the Combatant to it: `total_stat(stat)` now folds gear onto the base attribute, `derived_stat(stat, base)` applies gear to a base that lives elsewhere (per-attack damage), and `equipment_modifiers()` exposes the raw aggregate dict for sheets/tooltips. Removed the Phase 2 `fold_stat` placeholder (EffectResolver replaces it). NOTE: combatant.gd references `EffectResolver` by its global class_name — when I first saved, the editor flagged "EffectResolver not declared" until it rescanned the new file (same new-class_name lag as Combatant in Phase 2; clears on focus/reload).

### Phase 4 — leveling & progression ###
20. Filled the two 0-byte level_ups stubs. `level_value_database.gd` is now a loader for a new `level_curve` DbLoader category, reading `JSON/default_curve.json` — a per-level table of `{ level, xp_to_reach (total accumulated XP), gains (stat deltas) }`. Shipped a public default curve (levels 1–5); the private per-character curves stay in the gitignored `level_curves/` folder. Numbers in JSON, math in code, as the plan wants.
21. Loader gotcha I deliberately avoided: keyed the curve by level as an INT in the loader (not via JsonDB.index_by, which keys by the raw JSON number — a float). In GDScript `curve[2]` and `curve[2.0]` are different keys, so index_by would have made every integer level lookup silently miss. Wrote a tiny custom get_all() that casts.
22. `level_ups.gd` is now `class_name Leveling` (static): `grant_xp(combatant, amount, curve?)` adds XP and applies every level-up the new total crosses (so one big reward can jump multiple levels), returning the levels gained; plus `level_up` (one-level primitive), `level_for_xp`, `xp_to_reach`, `max_level`. The curve is an optional arg — omit it to use the `level_curve` category, or pass a private per-character curve.
23. Combatant gained an `xp` field and `gain_level(gains)`: `max_health`/`max_mana` grow the pool and `current_*` rises by the same delta (no surprise full-heal or chip damage — documented that heal-on-level is a one-liner if I want it later); any other key grows that BASE attribute, which means Phase 3 gear modifiers still stack on top via total_stat. Kept the split clean: Leveling owns the curve/threshold math, Combatant owns how gains mutate its own fields.
24. Big win on testability vs Phases 2–3: because Leveling takes an injectable curve and gain_level is pure mutation, the WHOLE leveling flow runs without the DbLoader autoload. So `tools/combatant_test.gd` now covers it for real (inject a small curve, grant XP, assert level/strength/HP grew and that XP past the top grants nothing). The runtime demo (`node_2dtestdb.gd`, F5) shows Bram leveling on the real default curve, including a +400 XP grant that crosses levels 4 AND 5 at once. Re-ran the Node sim against the actual default_curve.json: 120→Lv2, 320→Lv3, 720→Lv4+5, str 10→17, maxHP 100→152 — all matched.
